# 数据库迁移系统

## 概述
本项目使用自定义的数据库迁移系统，提供迁移版本管理、执行、回滚等功能。

## 目录结构

```
migrations/
├── 001_init_multiuser_schema/
│   ├── up.sql        # 迁移 SQL
│   └── down.sql      # 回滚 SQL
├── 001_init_multiuser_schema.json  # 迁移定义
├── 002_init_business_tables/
│   ├── up.sql
│   └── down.sql
├── 002_init_business_tables.json
├── 003_link_students_to_users/
│   ├── up.sql
│   └── down.sql
└── 003_link_students_to_users.json
```

## 迁移系统功能

### 1. 迁移历史追踪
- 自动记录所有已执行的迁移
- 记录执行时间、耗时、状态
- 保存迁移脚本校验和

### 2. 迁移依赖管理
- 支持定义迁移间的依赖关系
- 自动检查依赖是否满足
- 按正确顺序执行迁移

### 3. 迁移回滚
- 支持回滚到指定迁移
- 自动执行回滚脚本
- 记录回滚历史

### 4. 并发控制
- 使用迁移锁防止并发执行
- 自动检测和释放锁
- 防止数据不一致

## 使用方法

### 初始化迁移系统

首次使用时，需要创建迁移管理表：

```bash
python -c "from src.storage.database.migration_manager import MigrationManager; MigrationManager()"
```

或者手动执行 SQL 脚本：

```bash
psql -U your_user -d your_database -f scripts/create_migration_tables.sql
```

### 查看迁移状态

```bash
python scripts/migrate.py status
```

输出示例：
```
======================================================================
数据库迁移状态
======================================================================
已执行的迁移数量: 0

还没有执行任何迁移
======================================================================
```

### 执行迁移

```bash
python scripts/migrate.py migrate
```

执行所有待执行的迁移。

### 回滚迁移

```bash
python scripts/migrate.py rollback <migration_id>
```

回滚指定的迁移。

### 创建新迁移

```bash
python scripts/migrate.py init \
  --migration-id 004_add_example_table \
  --migration-name "添加示例表" \
  --version "1.0.0" \
  --description "创建一个示例表用于测试"
```

这会创建以下文件：
- `migrations/004_add_example_table.json` - 迁移定义
- `migrations/004_add_example_table/up.sql` - 迁移 SQL 模板
- `migrations/004_add_example_table/down.sql` - 回滚 SQL 模板

## 迁移开发规范

### 1. 迁移 ID 命名规则

使用三位数字编号 + 描述性名称：

```
001_init_multiuser_schema
002_init_business_tables
003_link_students_to_users
```

### 2. 迁移定义文件

```json
{
  "migration_id": "004_add_example_table",
  "migration_name": "添加示例表",
  "version": "1.0.0",
  "description": "创建一个示例表用于测试",
  "dependencies": ["001_init_multiuser_schema"]
}
```

字段说明：
- `migration_id`: 迁移唯一标识
- `migration_name`: 迁移名称
- `version`: 迁移版本号
- `description`: 迁移描述
- `dependencies`: 依赖的迁移ID列表

### 3. 迁移 SQL (up.sql)

```sql
-- ============================================
-- 迁移: 添加示例表
-- ID: 004_add_example_table
-- 版本: 1.0.0
-- 描述: 创建一个示例表用于测试
-- ============================================

CREATE TABLE IF NOT EXISTS example_table (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 添加注释
COMMENT ON TABLE example_table IS '示例表';
COMMENT ON COLUMN example_table.name IS '名称';
```

### 4. 回滚 SQL (down.sql)

```sql
-- ============================================
-- 回滚: 添加示例表
-- ID: 004_add_example_table
-- ============================================

DROP TABLE IF EXISTS example_table CASCADE;
```

## 最佳实践

### 1. 原子性
- 每个迁移应该是原子的，要么全部成功，要么全部失败
- 使用事务确保数据一致性

### 2. 可回滚性
- 每个迁移都应该提供回滚脚本
- 回滚脚本应该完全撤销 up.sql 的所有操作

### 3. 幂等性
- 迁移脚本应该是幂等的，可以安全地重复执行
- 使用 `IF NOT EXISTS` 和 `ON CONFLICT` 等语法

### 4. 向后兼容
- 尽量保持向后兼容
- 避免破坏性变更
- 如需破坏性变更，应该提供数据迁移方案

### 5. 测试
- 在测试环境中先验证迁移脚本
- 测试 up 和 down 脚本
- 验证迁移后的数据正确性

## 迁移状态

| 状态 | 说明 |
|------|------|
| `success` | 迁移成功执行 |
| `failed` | 迁移执行失败 |
| `rolled_back` | 迁移已回滚 |

## 迁移历史表

迁移系统使用 `db_migrations.schema_migrations` 表记录迁移历史：

```sql
SELECT * FROM db_migrations.schema_migrations ORDER BY executed_at DESC;
```

字段说明：
- `migration_id`: 迁移ID
- `migration_name`: 迁移名称
- `version`: 迁移版本
- `description`: 迁移描述
- `checksum`: 脚本校验和（SHA256）
- `executed_at`: 执行时间
- `execution_time_ms`: 执行耗时（毫秒）
- `status`: 执行状态
- `rollback_at`: 回滚时间
- `error_message`: 错误信息

## 现有迁移列表

### 001_init_multiuser_schema
- **名称**: 初始化多用户架构
- **描述**: 创建 auth 和 memory schema，包括用户表、权限表、会话表、记忆表等
- **依赖**: 无

### 002_init_business_tables
- **名称**: 初始化业务表
- **描述**: 创建业务相关的数据表（students, courses, homeworks, coursewares, exercises, achievements）
- **依赖**: 001_init_multiuser_schema

### 003_link_students_to_users
- **名称**: 关联 students 表到 auth.users
- **描述**: 为 students 表添加 user_id 字段和外键约束，并将现有数据关联到 auth.users
- **依赖**: 001_init_multiuser_schema, 002_init_business_tables

## 故障排查

### 1. 迁移卡住
如果迁移过程卡住，可能是迁移锁未释放：

```sql
SELECT * FROM db_migrations.migration_lock;
```

手动释放锁：

```sql
UPDATE db_migrations.migration_lock
SET locked_at = NULL, locked_by = NULL, lock_reason = NULL
WHERE id = 1;
```

### 2. 迁移失败
查看迁移错误信息：

```sql
SELECT * FROM db_migrations.schema_migrations
WHERE status = 'failed'
ORDER BY executed_at DESC;
```

### 3. 数据不一致
如果迁移导致数据不一致，可以：
1. 回滚失败的迁移
2. 手动修复数据
3. 重新执行迁移

## 注意事项

1. **备份数据**: 在执行迁移前，建议先备份数据库
2. **测试环境**: 先在测试环境验证迁移脚本
3. **生产环境**: 生产环境迁移应在低峰期进行
4. **监控**: 迁移过程中密切监控数据库状态
5. **日志**: 保留迁移日志以备排查问题

## 迁移与 Alembic 的对比

本项目使用自定义迁移系统而非 Alembic 的原因：
1. 更轻量级，无需额外依赖
2. 更灵活，可以完全自定义
3. 适应现有架构，无需大规模重构
4. 学习成本低，团队易于维护

如果未来需要更强大的功能，可以考虑迁移到 Alembic。
