# 环境变量命名更新说明

## 📝 更新日期
2024

## 🎯 更新目标

将环境变量命名从特定平台前缀（COZE_）改为标准命名规范，提高兼容性和可维护性。

---

## 🔄 变更内容

### 旧的环境变量命名（已弃用）

```bash
COZE_WORKLOAD_IDENTITY_API_KEY=...
COZE_INTEGRATION_MODEL_BASE_URL=...
```

### 新的环境变量命名（推荐）

```bash
DASHSCOPE_API_KEY=...
OPENAI_BASE_URL=...
```

## 🔧 兼容性处理

为了保证向后兼容性，代码中实现了环境变量优先级机制：

```python
# 优先级：配置文件 > 新环境变量 > 旧环境变量
api_key = (
    api_config.get('api_key') or
    os.getenv("DASHSCOPE_API_KEY") or
    os.getenv("COZE_WORKLOAD_IDENTITY_API_KEY")
)
base_url = (
    api_config.get('base_url') or
    os.getenv("OPENAI_BASE_URL") or
    os.getenv("COZE_INTEGRATION_MODEL_BASE_URL")
)
```

这意味着：
1. 优先使用配置文件中的值
2. 如果配置文件中没有，则使用新的环境变量命名
3. 如果新环境变量也没有，则回退到旧的环境变量命名

## 🎯 变更原因

### 1. 标准化命名
- 使用行业通用的环境变量命名规范
- `DASHSCOPE_API_KEY` 是阿里云百炼（DashScope）的标准命名
- `OPENAI_BASE_URL` 是 OpenAI 兼容接口的标准命名

### 2. 提高兼容性
- 避免使用特定平台的前缀（如 COZE_）
- 便于在不同部署平台之间迁移
- 不绑定到任何特定的云服务提供商

### 3. 遵循最佳实践
- 环境变量名称应该描述功能，而非平台
- 使用通用的、标准化的命名
- 便于理解和维护

### 4. 避免冲突
- 特定平台的前缀可能与其他服务冲突
- 标准命名减少命名空间污染

---

## ⚠️ 注意事项

### 1. 旧命名不会被立即移除
- 为了向后兼容，代码仍支持旧的环境变量命名
- 但新部署应该使用新的命名
- 旧命名将在未来的版本中被标记为弃用

### 2. 配置文件优先
- 配置文件（`config/agent_llm_config.json`）中的配置优先级最高
- 如果配置文件中有值，环境变量会被忽略
- 建议优先使用配置文件，环境变量作为覆盖

### 3. 安全性
- 环境变量名称的变更不影响安全性
- API Key 的值本身没有变化
- 只需要修改环境变量的名称

## 🆘 常见问题

### Q1: 我已经在使用旧的环境变量，需要立即修改吗？

**A**: 不需要立即修改。代码仍支持旧的环境变量命名。但建议在下次部署时迁移到新的命名。

### Q2: 如果同时配置了新旧两种环境变量，会使用哪一个？

**A**: 会使用新的环境变量命名。优先级是：配置文件 > 新环境变量 > 旧环境变量。

### Q3: 配置文件和环境变量，哪个优先？

**A**: 配置文件优先。如果配置文件中配置了 API Key 和 Base URL，环境变量会被忽略。

### Q4: 旧的环境变量命名什么时候会被移除？

**A**: 旧命名将在未来的版本中被标记为弃用，但不会立即移除。我们会提前发布公告，给用户充足的时间迁移。
